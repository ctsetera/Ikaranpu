# リファクタリング進捗状況（v1.0.0）

## 1. 初回調査で洗い出した問題

### 最優先で対応したい問題

| 問題                                                                    | 修正すべき理由                                          | 現在の状況 |
|-----------------------------------------------------------------------|--------------------------------------------------|-------|
| バリデーション処理がAdd/EditのViewModelに重複しており、テストが不足している                        | 仕様変更時に画面ごとの差異や回帰が発生しやすい                          | 解決済み  |
| Add/Editで入力状態・復元処理が重複している                                             | 同じ修正を複数箇所へ反映する必要があり、SavedStateHandle復元時の不具合につながる | 解決済み  |
| 編集画面の復元値がDBから再取得した値で上書きされる                                            | プロセス再生成後にユーザーの編集中データを失う可能性がある                    | 解決済み  |
| `VoiceRepository`が`CancellationException`を通常エラーへ変換する                  | キャンセルが上位へ伝播せず、キャンセル後にエラー表示や後続処理が走る可能性がある         | 解決済み  |
| `AudioPlayerManager`が独自の`CoroutineScope`を生成し、`play()`が再生ループ開始直後に成功を返す | 呼び出し元のライフサイクルから処理が外れ、再生中の失敗を呼び出し元へ返せない           | 解決済み  |

### 既知のUI課題

| 問題                                                 | 修正すべき理由                      | 現在の状況 |
|----------------------------------------------------|------------------------------|-------|
| UI表示文字列がKotlinコードへハードコードされている                      | 文言管理が分散し、再利用・検索・Lint対応が難しい   | 解決済み  |
| TrackAdd/TrackEdit画面のComposableが重複している             | UI変更時の二重修正と表示差異につながる         | 解決済み  |
| 各画面で同様の`Scaffold`・`TopAppBar`を構築している               | 共通スタイルやナビゲーションボタンの変更が分散する    | 解決済み  |
| TrackEditor固有Composableの配置と公開範囲が不明瞭                | 汎用Componentと画面固有UIの責務が分かりにくい | 解決済み  |
| 一覧・下書き・再生画面でToastイベント購読処理が重複している                   | 副作用処理の修正が複数画面へ分散する           | 解決済み  |
| `collectAsState()`を使用しており、ライフサイクルに応じた収集停止が明示されていない | 画面が非表示でもFlow収集が継続する可能性がある    | 解決済み  |

### データ・非同期処理の問題

| 問題                                                         | 修正すべき理由                           | 現在の状況 |
|------------------------------------------------------------|-----------------------------------|-------|
| Add/Update UseCaseで音声生成・進捗通知・保存処理が重複している                   | エラー処理や進捗仕様の変更時に差異が生まれやすい          | 解決済み  |
| VOICEVOX生成状況のポーリングにタイムアウトがない                               | APIが準備完了を返さない場合、キャンセルされるまで処理が継続する | 解決済み  |

| `TrackRepository`が空一覧を`TrackNotFound`エラーとして返す              | 正常な空状態とデータ取得失敗をUI側で区別しにくい         | 解決済み  |
| `MainActivity`内でRepository・UseCase・ViewModel生成が重複している      | 依存関係の共有、差し替え、単体テストが難しい            | 解決済み  |
| Repository・UseCase・非同期処理のテストが不足している                        | キャンセル、API失敗、DB失敗など重要な分岐を保証できない    | 解決済み  |
